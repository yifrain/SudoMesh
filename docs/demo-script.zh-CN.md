## 第 0 部分 — 立意 + 架构  *（主讲：E）*

**[说]**"SwarmSolve 在去中心化 P2P 网络上求解超大数独（最大 25×25）——没有中央服务器。
数独是一棵巨大的搜索树；我们把它切成子任务，许多地位平等的节点并行探索不同部分，彼此交换
三类消息：**Open Task（开放任务）**、**Dead End（死路）**、**Solution（解）**。"

**[看]** `docs/architecture.zh-CN.md` 第 1 节的五层架构图。自底向上点名各层 + 一句课程映射：

> 传输（第2章）→ **发现：Kademlia DHT（第6章）** → 传播（第2章+第7章）→
> 任务：切分/去重/租约（第5章）→ 求解。

**[亮点]** 一句话点出最大亮点：*"我们把 Kademlia 的 XOR 键空间复用为任务 ID 空间,
所以 DHT 同时充当负载均衡器——这把第6章和第5章直接打通了。"*

## 第 1 部分 — 单机求解基线  *（主讲：D）*

**[说]**"先看单机。求解器用位掩码约束传播 + MRV 深度优先搜索。"

**[跑]**
```bash
uv run swarmsolve solve examples/puzzles/hard_9x9.txt
```
**[看]** `Solved!`、棋盘，以及类似 `time=0.1s nodes=6050` 的一行。

**[亮点]**"6050 个节点是单机的工作量。记住它——集群将把这类工作切分到各节点。"

---

## 第 2 部分 — 真实 P2P 集群 + 正确性  *（主讲：C）*

**[说]**"现在是真家伙：四个**独立的操作系统进程**，通过本机真实 TCP/UDP 通信。一个节点提交；
所有节点经 Kademlia 自发现、gossip 开放任务、用租约认领、共享死路，第一个解让所有节点停止。"

**[跑]**
```bash
uv run swarmsolve demo --file examples/puzzles/hard_9x9.txt --peers 4
```
**[看]** 每节点报告表，然后 `Swarm solved!` 和棋盘。

**[亮点]**"这是真实进程、真实 socket——不是线程。指着每节点的 `nodes`/`tasks_done` 列：
工作确实被分发了。每个节点都拿到解，是因为解经 gossip 传播。"

> 主动说出的诚实声明："在很小的 9×9 上集群并不比单机快——协调开销超过搜索本身。
> 真正的加速体现在穷举搜索上，我会在第 5 部分演示。"

---

## 第 3 部分 — 容错  *（主讲：E）*  ⭐ 重点

**[说]**"节点会崩溃。我们用**有时限的租约**：节点认领任务时持有租约；若它死了，租约过期，
另一个节点接管该任务。我用穷举模式跑，使*每个*任务都必须完成——也就是说只有重分配真正生效，
整个运行才能结束。然后我在求解中途杀掉一个节点。"

**[跑]**
```bash
uv run swarmsolve fault --file examples/puzzles/hard_9x9.txt \
    --peers 4 --kill-peer 2 --kill-after 1.5 --lease 1.5 --node-delay 0.0008 --split-depth 4
```
**[看]**
```
>>> killed peer #2 (PID …)
   killed peer #2 returned a result: no (as expected)
   surviving peers that finished: [0, 1, 3]
   swarm STILL solved the puzzle in ~13s despite the failure
```
**[亮点]**"节点 2 再也没返回——但集群仍然完成，因为它租约下的任务被存活者接管了。
这就是**基于租约 + `reclaim_expired` 的容错**，再加上 Kademlia 的抗抖动（k桶绕过失效联系人）。"

**[说]**"为了让集群可观测，每个节点上报快照，我们渲染一张实时表。"

**[亮点]**"可以看到任务在 open/claimed/done 列之间流动、死路数随剪枝传播而增长——协议在运转。"

## 第 5 部分 — 诚实加速 + scaling  *（主讲：B）*  ⭐ 重点

**[说]**"首解搜索几乎无法并行——答案在一条深 DFS 路径上。真正可并行的负载是**穷举搜索**：
统计所有解 / 验证唯一性。我们用**精确模式**跑：每个任务有唯一负责人（XOR 最近的节点），经 TCP
可靠投递，并用**虚拟节点**均衡负载——于是零重复工作、解计数精确。"

**[跑]**（先单跑 4 节点）
```bash
uv run swarmsolve benchmark --file wide.txt --peers 4 --node-delay 0.0002
```
**[看]** 各节点节点数均衡（每个约 24k），然后：
```
correctness OK: all 45475 solution(s) covered exactly once
speedup  : 2.55x (wall clock)
```
**[说]**"而且它能 scaling——这是 1、2、4 节点："

**[跑]**（时间够再跑）
```bash
for p in 1 2 4; do uv run swarmsolve benchmark --file wide.txt --peers $p --node-delay 0.0002; done
```
**[看]** `docs/architecture.zh-CN.md` 第 14 节的加速曲线图：

| 节点数 | 墙钟 | vs 基线 | vs 单节点 |
|--------|------|---------|-----------|
| 1 | 34.8 s | 0.80× | 1.0× |
| 2 | 20.3 s | 1.50× | 1.71× |
| 4 | 11.6 s | 2.55× | **3.0×** |

**[亮点]**"相对单节点，4 节点约 3×——**75% 并行效率**——且**解计数精确、约 0% 重复工作**。
虚拟节点（一致性哈希，第5章）是负载均衡的关键——没有它，某个节点要做 47% 的工作。"

## 第 6 部分 — 设计权衡  *（主讲：A）*

**[说]**"我们支持两种执行模式，二者的对比正是有意思的地方："

| 模式 | 重复工作 | 计数 | 擅长 | 鲁棒性 |
|------|----------|------|------|--------|
| 工作窃取 | 有（节点级） | 精确（dedup） | 不平衡树 | 容忍丢失/抖动 |
| 精确（默认） | 无 | 精确 | 平衡树 | 需要可靠投递 |

**[亮点]**"这是一个干净的**一致性 vs 可用性**权衡：精确模式偏向精确、零重复；工作窃取偏向
抖动下的可用性与吞吐。另外两个有意的选择：**发现用 Kademlia/UDP、任务用 TCP**（满足作业的
'TCP 消息'传可靠负载），以及**死路按深度限制共享**，使难题不会用上万条消息淹没网络。"

## 课程知识点（自然地穿插点出）

* **第2章** — gossip + TTL + seen 去重（修正 Gnutella 的泛洪）。
* **第5章** — 经 XOR 均匀放置 + 虚拟节点实现负载均衡。
* **第6章** — Kademlia：XOR 度量、k桶、迭代 FIND_NODE。
* **第7章** — BubbleStorm 式概率覆盖（gossip 扇出高概率覆盖全网）。

## 预设问答

**问：为什么 9×9 集群比单机慢？**
答：首解搜索沿解路径本质上是串行的；小树上协调开销占主导。并行收益在穷举搜索——见第 5 部分
（4 节点 2.55×）。

**问：如何避免两个节点做同样的工作？**
答：两层。任务在调度器里按 ID 去重，认领前还有最后一刻检查。精确模式下每个任务有唯一的
XOR-最近负责人、经 TCP 可靠投递，所以重复约 0%（已验证：节点总数与基线一致、解计数精确）。

**问：节点加入或离开时会怎样？**
答：加入：Kademlia bootstrap + FIND_NODE 填充 k桶。离开/崩溃：其任务租约过期，存活者接管
（第 3 部分）。归属基于 XOR 距离，所以平滑迁移。

**问：精确模式传了 roster，还算去中心化吗？**
答：roster 只是为了干净的本机基准；同样的归属可由 Kademlia 路由表经 `is_responsible_for`
计算（不传 roster 时即默认如此）。发现、gossip、租约都是完全去中心化的。

**问：为什么用 Kademlia 而非 Chord/CAN？**
答：Kademlia 的 XOR 度量对称，让同一个键空间同时服务路由与任务放置；k桶偏好长寿节点（抗日食）；
基于 UDP 轻量、抗抖动——正是 BitTorrent/IPFS 的用法。

**问：能扩展到 25×25 吗？**
答：求解器与尺寸无关（N=k²）；`gen --size 25` 瞬时生成，高线索实例靠约束传播立即解出。
稀疏 25×25 首解是 NP 难（树爆炸）——这是数独的性质，不是我们系统的问题；分布式加速用穷举基准量化。

**问：下一步做什么？**
答：无需 roster 的抖动感知归属、针对不平衡树的自适应工作捐赠、以及拼图扩展（只换 `solver/` 包）。

## 万一命令卡住或端口被占

* Ctrl-C，然后重跑；端口一两秒内释放。
* 把 `--peers` 降到 3，或机器慢就把 `--node-delay` 再调小 2 倍。
* 兜底讲法：单元测试（`uv run pytest -q`）无需任何网络即可证明求解器 + DHT 的正确性。
